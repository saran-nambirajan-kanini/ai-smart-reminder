# Architecture Design

## Project Overview

AI Smart Reminder is an AI-powered to-do list and reminder application that augments traditional task management with intelligent scheduling, smart prioritization, natural language task creation, and personalized productivity insights. The system targets working professionals, students, freelancers, and small teams who need a differentiated task management experience. It delivers a responsive Angular web application backed by a .NET Web API with PostgreSQL persistence and OpenAI/Azure OpenAI integration for AI-powered features including NLP parsing, auto-categorization, priority suggestions, and analytics.

## Architecture Goals

- Architecture Goal 1: Separation of Concerns — Isolate domain logic from infrastructure dependencies using Clean Architecture with explicit port/adapter boundaries to enable independent testing and provider swapping
- Architecture Goal 2: API-First Design — Expose all functionality through a RESTful API with OpenAPI documentation to support the Angular SPA and enable future mobile client integration without backend changes
- Architecture Goal 3: AI Decoupling — Abstract AI features behind service interfaces with circuit breaker patterns and deterministic fallback paths so the system remains fully functional when AI services are unavailable
- Architecture Goal 4: Scalability — Design stateless API tier with distributed background job processing to support horizontal scaling for concurrent users and reminder processing load
- Architecture Goal 5: Security by Design — Implement OWASP-aligned security controls including JWT authentication, row-level data isolation, input validation, encryption, and rate limiting as first-class architectural concerns
- Architecture Goal 6: Observability — Integrate structured logging with correlation IDs, health check endpoints, and metrics collection to enable proactive monitoring and rapid incident response

## Non-Functional Requirements

### Performance

- NFR-001: System MUST respond to all API requests within 3 seconds under normal load (≤ 1,000 concurrent users)
- NFR-002: System MUST refresh dashboard analytics data at minimum every 5 minutes via background aggregation jobs
- NFR-003: System MUST trigger scheduled reminders within a 1-minute accuracy window of the configured reminder time

### Availability

- NFR-004: System MUST maintain 99.9% uptime (≤ 8.76 hours unplanned downtime per year) excluding scheduled maintenance windows

### Scalability

- NFR-005: System MUST support 10,000 concurrent authenticated users without degradation below NFR-001 response time thresholds
- NFR-006: System MUST handle storage and retrieval of 1 million tasks and 500,000 active reminders without query performance degradation

### Security

- NFR-007: System MUST encrypt all data in transit using TLS 1.2 or higher and encrypt sensitive data at rest using AES-256 encryption
- NFR-008: System MUST hash passwords using bcrypt (work factor ≥ 12) or Argon2id, enforcing minimum 8 characters with uppercase, lowercase, number, and special character requirements
- NFR-009: System MUST issue JWT access tokens with a maximum expiry of 15 minutes and implement refresh token rotation with single-use enforcement
- NFR-010: System MUST enforce rate limiting on all API endpoints (default: 100 requests per minute per authenticated user, 20 requests per minute for unauthenticated endpoints)
- NFR-011: System MUST validate and sanitize all user inputs at the API boundary to prevent injection attacks (SQL injection, XSS, command injection)

### Reliability

- NFR-012: System MUST retry failed notification deliveries up to 3 times with exponential backoff (1s, 4s, 16s) before marking as failed, and attempt a fallback notification channel if the primary channel fails
- NFR-013: System MUST implement circuit breaker pattern for all external service calls (AI provider, notification services, OAuth providers) with configurable failure thresholds (default: 5 failures in 60 seconds opens circuit for 30 seconds)

### Maintainability

- NFR-014: System MUST implement Clean Architecture with defined layer boundaries (Domain, Application, Infrastructure, Presentation) where inner layers have zero dependencies on outer layers
- NFR-015: System MUST achieve minimum 80% unit test code coverage for Domain and Application layers, and minimum 60% integration test coverage for Infrastructure layer

### Usability

- NFR-016: System MUST provide a responsive user interface that renders correctly on viewports from 320px (mobile) to 2560px (desktop) with no horizontal scrolling or layout breakage

### Observability

- NFR-017: System MUST emit structured log entries (JSON format) with correlation IDs for all API requests, background jobs, and external service calls to enable end-to-end request tracing
- NFR-018: System MUST expose health check endpoints for all external dependencies (database, Redis, AI provider, notification services) returning standardized health status responses

## Data Requirements

### Data Structure

- DR-001: System MUST store user profiles with email as unique identifier, supporting name, hashed password, notification preferences (per-channel enable/disable and quiet hours), role (User/Admin), OAuth provider linkage, and active status
- DR-002: System MUST store tasks with a unique identifier, owner reference (UserId), title (required), description, due date, priority (enum: High/Medium/Low, default Medium), status (enum: Active/Completed/Deleted), completion timestamp, soft-delete flag with deletion timestamp, and audit timestamps (CreatedAt, UpdatedAt)
- DR-003: System MUST store reminders with a unique identifier, associated task reference, owner reference, scheduled time (UTC), frequency type (enum: None/Daily/Weekly/Monthly/Custom), custom interval in days, active status, last triggered timestamp, next occurrence timestamp, and snooze-until timestamp
- DR-004: System MUST store notifications with a unique identifier, recipient reference, associated reminder reference, delivery channel (enum: Push/Email/SMS), message content, delivery status (enum: Pending/Sent/Delivered/Failed), read status, sent timestamp, and creation timestamp
- DR-005: System MUST store audit log entries with a unique identifier, actor reference (UserId), action type, target entity type and identifier, action details (JSON), source IP address, and UTC timestamp
- DR-006: System MUST store user-defined categories with a unique identifier, owner reference, and category name, supporting many-to-many relationships with tasks via a join entity (TaskCategory)
- DR-007: System MUST store task notes with a unique identifier, parent task reference, text content, and audit timestamps (CreatedAt, UpdatedAt)

### Data Integrity

- DR-008: System MUST enforce referential integrity between all related entities (User→Task, Task→Reminder, Task→Note, Reminder→Notification, User→Category, Task↔Category) using foreign key constraints with appropriate cascade rules
- DR-009: System MUST store all date and time values in UTC format in the database, with timezone conversion performed exclusively on the client side using the IANA timezone database
- DR-010: System MUST enforce unique constraints on user email addresses and prevent duplicate category names per user

### Data Retention

- DR-011: System MUST retain soft-deleted tasks for a minimum of 12 months for analytics and audit purposes before eligible for permanent deletion
- DR-012: System MUST retain audit log entries for a minimum of 24 months with no modification or deletion permitted during the retention period

### Data Backup

- DR-013: System MUST perform automated daily database backups with point-in-time recovery capability within a 30-day retention window
- DR-014: System MUST support database restoration to any point-in-time within the backup retention window with a recovery time objective (RTO) of 4 hours

### Data Migration

- DR-015: System MUST manage database schema changes exclusively through Entity Framework Core migrations with version tracking
- DR-016: System MUST support zero-downtime schema migrations by ensuring all migrations are backward-compatible with the previous application version

### Domain Entities

- **User**: Represents an authenticated end user or administrator. Attributes: Id (GUID), Name, Email (unique), PasswordHash, NotificationPreferences (JSON), QuietHoursStart, QuietHoursEnd, Role (User/Admin), OAuthProvider, OAuthId, IsActive, CreatedAt, UpdatedAt. Relationships: owns many Tasks, Categories, Reminders, Notifications, AuditLogs
- **Task**: Core work item managed by a user. Attributes: Id (GUID), UserId (FK), Title, Description, DueDate, Priority (High/Medium/Low), Status (Active/Completed/Deleted), CompletionTimestamp, IsDeleted, DeletedAt, CreatedAt, UpdatedAt. Relationships: belongs to User, has many Notes, has many Reminders, has many Categories (M:N via TaskCategory)
- **Category**: User-defined classification label for tasks. Attributes: Id (GUID), UserId (FK), Name. Relationships: belongs to User, has many Tasks (M:N via TaskCategory)
- **TaskCategory**: Join entity for Task-Category many-to-many relationship. Attributes: TaskId (FK), CategoryId (FK)
- **Note**: Text annotation attached to a task. Attributes: Id (GUID), TaskId (FK), Content, CreatedAt, UpdatedAt. Relationships: belongs to Task
- **Reminder**: Scheduled trigger for task notifications. Attributes: Id (GUID), TaskId (FK), UserId (FK), ScheduledTime (UTC), FrequencyType (None/Daily/Weekly/Monthly/Custom), CustomIntervalDays, IsActive, LastTriggeredAt, NextOccurrence, SnoozedUntil, CreatedAt, UpdatedAt. Relationships: belongs to Task, belongs to User, has many Notifications
- **Notification**: Delivery record for a reminder notification. Attributes: Id (GUID), UserId (FK), ReminderId (FK), Channel (Push/Email/SMS), Content, DeliveryStatus (Pending/Sent/Delivered/Failed), IsRead, SentAt, CreatedAt. Relationships: belongs to User, belongs to Reminder
- **AuditLog**: Immutable record of user actions for compliance tracking. Attributes: Id (GUID), UserId (FK), Action, EntityType, EntityId, Details (JSON), IpAddress, Timestamp (UTC). Relationships: belongs to User
- **RefreshToken**: JWT refresh token management entity. Attributes: Id (GUID), UserId (FK), Token (hashed), ExpiresAt, RevokedAt, CreatedAt. Relationships: belongs to User
- **UserProductivityMetrics**: Pre-aggregated analytics data for dashboard display. Attributes: Id (GUID), UserId (FK), PeriodStart, PeriodEnd, PeriodType (Daily/Weekly/Monthly), TotalTasks, CompletedTasks, MissedReminders, OnTimeCompletionRate, ProductivityScore, MostProductiveDay, AverageCompletionTime, CreatedAt. Relationships: belongs to User
- **AIInteractionLog**: Audit trail for AI API invocations. Attributes: Id (GUID), UserId (FK), RequestType, PromptSummary, ResponseSummary, TokensUsed, LatencyMs, ModelVersion, CreatedAt. Relationships: belongs to User

### AI Functional Requirements

### AI Architecture Pattern

**Selected Pattern:** Tool Calling + Direct Prompting (Hybrid)

**Rationale:** The AI features in this application operate on structured data (tasks, user history, dates) rather than unstructured document corpora, eliminating the need for RAG. NLP parsing, classification, and insight generation are best served by direct LLM prompting with structured output (JSON mode) and few-shot examples. No fine-tuning is needed because general-purpose LLM capabilities are sufficient for task classification and date extraction; domain adaptation is achieved through system prompts and examples. The hybrid pattern combining direct prompting for NLU tasks with tool calling patterns for system integration (creating tasks, setting reminders from parsed data) provides the optimal balance of capability and simplicity.

**Pattern Selection Justification:**

| Pattern | Decision | Rationale |
|---------|----------|-----------|
| RAG | Not Selected | No document corpus to search; features use structured relational data + LLM reasoning |
| Tool Calling | Selected (partial) | Task creation and reminder setting from parsed NLP output requires system integration |
| Fine-tuning | Not Selected | General-purpose LLM capabilities sufficient; domain adaptation via system prompts |
| Direct Prompting | Selected (primary) | NLP parsing, classification, prioritization, and insights generation via structured prompting |

## Architecture and Design Decisions

- Decision 1: **Clean Architecture with 4 Layers** — Domain (entities, interfaces), Application (services, DTOs, use cases), Infrastructure (EF Core, external integrations), Presentation (Angular SPA, API controllers). Context: Multiple external dependencies (AI, notifications, OAuth) require isolation. Decision: Adopt Clean Architecture with strict dependency inversion. Benefit: Inner layers testable without infrastructure; providers swappable without domain changes. Driven by NFR-014.

- Decision 2: **API-First RESTful Design** — All frontend-backend communication via versioned RESTful API with OpenAPI/Swagger documentation. Context: Web-first MVP with future mobile client planned. Decision: Single API surface for all clients. Benefit: Mobile-ready API without backend changes; contract-first development enables parallel frontend/backend work.

- Decision 3: **CQRS-Lite for Analytics** — Separate pre-aggregated read models for dashboard analytics computed by background jobs. Standard repository pattern for write operations. Context: Real-time dashboard aggregations across large datasets would violate NFR-001 response times. Decision: Background jobs pre-compute analytics into UserProductivityMetrics table. Benefit: Dashboard queries hit pre-computed data, ensuring sub-second response. Driven by NFR-002.

- Decision 4: **AI Abstraction Layer with Strategy Pattern** — IAiService interface with concrete implementations (AzureOpenAiService, OpenAiService). Circuit breaker wrapping via Polly. Fallback to deterministic logic when AI is unavailable. Context: AI features depend on external API availability and incur per-token costs. Decision: Provider-agnostic interface with runtime configuration. Benefit: Swap AI providers without code changes; system remains functional during AI outages. Driven by AIR-004, AIR-O02.

- Decision 5: **Event-Driven Notification Pipeline** — Reminder triggers publish domain events; notification handlers process per-channel delivery asynchronously via background jobs. Context: Multi-channel delivery with retry logic must not block the reminder scheduling path. Decision: Decouple trigger from delivery via Hangfire background jobs. Benefit: Independent retry per channel; failure in one channel does not affect others. Driven by NFR-012.

- Decision 6: **Hangfire for Background Job Processing** — Hangfire with PostgreSQL storage for recurring reminder scheduling, notification delivery, analytics aggregation, and AI metric computation. Context: 1-minute accuracy for reminder triggers requires persistent, reliable scheduling. Decision: Hangfire provides cron-based scheduling, persistence, retries, and a monitoring dashboard. Benefit: Persistent jobs survive application restarts; built-in retry with backoff. Driven by NFR-003.

- Decision 7: **Redis Distributed Caching** — Redis as optional caching layer for JWT blacklisting, rate limiting counters, AI response caching, and frequently accessed user preferences. Context: Rate limiting and token revocation require shared state across API instances. Decision: Redis provides distributed cache with pub/sub. Benefit: Enables horizontal API scaling; AI response caching reduces token costs. Driven by NFR-010, NFR-005.

- Decision 8: **Row-Level Data Isolation** — EF Core global query filters applying UserId-based filtering on all user-owned entities. Admin role bypasses filters. Context: Multi-user application requires strict data isolation without multi-tenant infrastructure complexity. Decision: Automatic query filtering at ORM level. Benefit: Prevents data leakage between users; developers cannot accidentally query other users' data. Driven by NFR-011, OWASP A01.

- Decision 9: **UTC-First Temporal Design** — All timestamps stored as UTC in PostgreSQL (timestamptz). Client-side timezone conversion using IANA timezone database. Context: Reminders must trigger at correct times across timezones and DST transitions. Decision: UTC storage with client-side presentation conversion. Benefit: Eliminates server-side timezone ambiguity; DST transitions handled by proven client libraries. Addresses Risk: "Time zone and DST handling errors."

- Decision 10: **Soft-Delete with EF Core Global Filters** — Deleted tasks marked with IsDeleted flag and DeletedAt timestamp. EF Core query filters exclude soft-deleted records from standard queries. Analytics queries use unfiltered DbSet. Context: Tasks must be retained for analytics while appearing deleted to users. Decision: Shadow-style soft delete with automatic query exclusion. Benefit: Analytics data preserved; user experience shows clean deletion. Driven by DR-011, FR-010.

### Alternative Technology Options

- **Backend Alternative — Node.js (NestJS)**: Considered for its async I/O model and JavaScript ecosystem alignment with Angular frontend. Not selected because .NET 8 provides superior performance benchmarks for CPU-bound operations (AI response processing, analytics aggregation), stronger type safety with C#, and native Clean Architecture patterns. The spec explicitly mandates .NET Web API.
- **Database Alternative — SQL Server**: Considered for its tight .NET integration and Azure-managed service. Not selected because PostgreSQL offers equivalent EF Core support with superior JSON querying (jsonb), built-in full-text search without additional services, and lower licensing costs for self-hosted deployments. The spec explicitly mandates PostgreSQL.
- **AI Alternative — Anthropic Claude**: Considered for its strong reasoning capabilities and safety features. Not selected because Azure OpenAI provides enterprise compliance controls (data residency, private endpoints), same GPT-4o model quality, and better .NET SDK support via Azure.AI.OpenAI NuGet package.
- **Background Jobs Alternative — Quartz.NET**: Considered for its mature cron scheduling capabilities. Not selected because Hangfire provides a simpler API, built-in PostgreSQL storage without additional schema setup, a web-based monitoring dashboard, and automatic retry with configurable backoff.

## Technical Constraints & Assumptions

### Constraints

- Initial release targets web-only (Angular SPA); native mobile applications are deferred to a future phase. API-first design ensures mobile readiness.
- AI features depend on external API availability (Azure OpenAI / OpenAI) and incur per-token costs. Token budgeting (AIR-O01: 4,000 tokens per request) and response caching are required for cost control.
- PostgreSQL is the single source of truth. Redis is an optional caching layer that enhances performance but is not required for correctness. The system must function (with degraded performance) if Redis is unavailable.
- Password complexity requirements are fixed: minimum 8 characters, uppercase, lowercase, number, special character (NFR-008).
- JWT access token maximum expiry is 15 minutes with refresh token rotation (NFR-009). This is a security constraint that cannot be relaxed.
- Rate limiting defaults (100 req/min authenticated, 20 req/min unauthenticated) are configurable but must be enforced from initial deployment (NFR-010).

### Assumptions

- Users have reliable internet access; offline mode is not supported in the initial release. Offline support (sync, conflict resolution) is deferred to future phases.
- Email (SendGrid) and SMS (Twilio) notification providers are pre-configured and operational before deployment. Provider API keys and configuration are deployment prerequisites.
- Firebase Cloud Messaging is configured with appropriate web push certificates before deployment.
- The development team has working knowledge of C#/.NET, Angular/TypeScript, and PostgreSQL. No specialized ML/AI engineering expertise is required as AI features use managed LLM APIs via prompting (no model training or fine-tuning).
- Azure OpenAI service quotas are sufficient for projected usage (estimated 50,000 AI requests per day at scale). Quota increases may be required as user base grows.
- A single deployment region is sufficient for initial launch. Multi-region deployment is a future scalability consideration.

## Development Workflow

1. **Environment Setup**: Provision development environment with Docker Compose (PostgreSQL, Redis, Hangfire dashboard), configure Azure OpenAI API keys, set up Angular CLI and .NET SDK
2. **Domain Layer First**: Implement domain entities, value objects, enums (Priority, FrequencyType, DeliveryStatus), and domain interfaces (ITaskRepository, IReminderRepository, IAiService, INotificationService)
3. **Infrastructure Layer**: Implement EF Core DbContext with entity configurations, global query filters, migrations, and external service clients (AI provider, notification channels, OAuth)
4. **Application Layer**: Implement application services, DTOs, validators (FluentValidation), and mapping profiles (AutoMapper) orchestrating domain logic and infrastructure
5. **API Layer**: Implement controllers, middleware (authentication, error handling, correlation ID, rate limiting), and OpenAPI documentation
6. **Frontend Development**: Implement Angular application with Core module (auth, HTTP interceptors), Shared module (components, pipes), and Feature modules (tasks, reminders, notifications, dashboard, admin)
7. **Background Jobs**: Implement Hangfire jobs for reminder scheduling, notification delivery, analytics aggregation, and AI metric computation
8. **AI Integration**: Implement AI service layer with Azure OpenAI client, prompt templates, structured output parsing, circuit breaker, and fallback logic
9. **Testing**: Execute unit tests (Domain, Application layers), integration tests (Infrastructure, API), and end-to-end tests (Angular + API)
10. **Deployment**: Containerize with Docker, configure CI/CD pipeline, deploy to target environment with health check verification
