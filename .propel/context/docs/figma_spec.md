## 7. Content & Tone

### Voice & Tone

- **Overall Tone**: Professional, warm, and encouraging — a productivity companion that feels supportive without being patronizing
- **Error Messages**: Helpful, non-blaming, actionable (e.g., "We could not find that email address. Check the spelling or create a new account.")
- **Empty States**: Encouraging with clear call-to-action (e.g., "No tasks yet. Create your first task to get started!")
- **Success Messages**: Brief and celebratory (e.g., "Task created successfully!", "Reminder set for tomorrow at 6 PM")
- **AI Suggestions**: Transparent and explanatory (e.g., "Based on your deadline and past completion patterns, I suggest High priority")

### Content Guidelines

- **Headings**: Sentence case (e.g., "Upcoming reminders" not "Upcoming Reminders")
- **CTAs**: Action-oriented with specific verbs (e.g., "Create task", "Set reminder", "View analytics")
- **Labels**: Concise and descriptive (e.g., "Due date", "Priority level", "Notification channel")
- **Placeholder Text**: Helpful examples (e.g., "Remind me tomorrow at 6 PM to submit the project")
- **Dates**: Relative when recent ("2 hours ago", "Yesterday"), absolute when distant ("May 14, 2026")

## 8. Data & Edge Cases

### Data Scenarios

| Scenario | Description | Handling |
|----------|-------------|----------|
| No Data | User has zero tasks/reminders | Empty state illustration with CTA ("Create your first task") |
| First Use | New user, no completion history | Onboarding hint cards on dashboard; AI insights show "More data needed for full insights" |
| Large Data | 1,000+ tasks (per NFR-006: 1M tasks system-wide) | Server-side pagination (20 items/page); virtual scrolling for long lists |
| Slow Connection | API response > 3 seconds | Skeleton screens preserving layout; abort after 10 seconds with error state |
| Offline | No network connectivity | Banner: "You are offline. Some features are unavailable." Read-only cached data where possible |
| AI Unavailable | AI service down or circuit breaker open | Graceful fallback: manual form replaces NL input; deterministic sort replaces AI priority; metrics-only dashboard |

### Edge Cases

| Case | Screen(s) Affected | Solution |
|------|-------------------|----------|
| Long task title | SCR-006, SCR-007, SCR-012 | Truncate with ellipsis after 2 lines; full title in tooltip |
| Long description | SCR-007 | Collapsible section; "Show more" / "Show less" toggle |
| Missing due date | SCR-006, SCR-012 | Display "No due date" in muted text; sort to end when sorting by due date |
| Concurrent session | All authenticated | Toast: "Your session was updated in another tab"; refresh data |
| Session timeout | All authenticated | Modal overlay: "Your session has expired. Sign in to continue." with login form; preserve unsaved data in localStorage |
| Timezone mismatch | SCR-009, SCR-010 | Display times in user's local timezone (IANA); show timezone abbreviation on hover |
| Rapid form submission | All forms | Disable submit button after first click; re-enable on error |
| Overdue task | SCR-006, SCR-013 | Red badge/indicator; "Overdue" status tag; separate section on dashboard |

## 11. Prototype Flows

### Flow: FL-001 - Authentication

**Flow ID**: FL-001
**Derived From**: UC-001
**Personas Covered**: User, Admin
**Description**: Complete authentication flow covering email/password login, OAuth login, registration, and password reset

#### Flow Sequence

```text
1. Entry: SCR-001 (Login) / Default
   - Trigger: User navigates to application URL
   |
   v
2. Decision: Login method?
   +-- Email/Password:
   |   - Action: Enter credentials, click Login
   |   - Loading: SCR-001 / Loading (authenticating)
   |   +-- Success -> SCR-013 (Dashboard) / Default
   |   +-- Error -> SCR-001 / Error (invalid credentials)
   |   +-- Validation -> SCR-001 / Validation (field errors)
   |
   +-- OAuth (Google/Microsoft):
   |   - Action: Click OAuth provider button
   |   - External: OAuth provider login page
   |   +-- Success -> SCR-013 (Dashboard) / Default
   |   +-- Error -> SCR-001 / Error (OAuth failed)
   |
   +-- Register:
   |   - Action: Click "Create account" link
   |   -> SCR-002 (Registration) / Default
   |   - Action: Fill form, submit
   |   +-- Success -> SCR-001 (Login) / Default (registration complete message)
   |   +-- Validation -> SCR-002 / Validation (field errors)
   |   +-- Error -> SCR-002 / Error (email taken)
   |
   +-- Forgot Password:
       - Action: Click "Forgot password" link
       -> SCR-003 (Password Reset Request) / Default
       - Action: Enter email, submit
       +-- Success -> SCR-003 / Default (email sent confirmation)
       +-- Error -> SCR-003 / Error (email not found)
       -> (via email link) SCR-004 (Password Reset Confirm) / Default
       - Action: Enter new password, submit
       +-- Success -> SCR-001 (Login) / Default (password reset message)
       +-- Error -> SCR-004 / Error (link expired)
```

#### Required Interactions

- **Email field**: Auto-focus on page load; keyboard Enter advances to password
- **Password field**: Toggle visibility icon; keyboard Enter triggers login
- **OAuth buttons**: External redirect with loading indicator
- **Form submission**: Button disabled during loading; re-enabled on error

**Flow ID**: FL-004

**Description**: Set a reminder on a task, receive notification, and manage snooze

#### Flow Sequence

```text

   - Toast notification appears if user is in-app
   |

   +-- View -> SCR-015 (Notification History) / Default

```

- **Recurrence select**: Conditional field (custom interval) appears when "Custom" selected

### Flow: FL-009 - Error Recovery

**Flow ID**: FL-009
**Derived From**: UC-003 (5a), UC-004 (2a), UC-005 (2a)
**Personas Covered**: User, Admin
**Description**: Generic error recovery patterns for network failures, API errors, and AI service unavailability

#### Flow Sequence

```text
1. Entry: Any screen / Error state
   - Trigger: API call fails or times out
   |
   v
2. Step: Error state displayed
   - Error message explains the issue
   - Primary action: "Try again" (retry button)
   - Secondary action: Context-specific fallback
   |
   v
3. Decision: Retry?
   +-- Retry -> Loading state -> Success (return to Default) or Error (remain)

```

#### Required Interactions

- **Retry button**: Visible and prominent in all error states
- **Auto-retry**: For notification delivery failures (transparent to user; status badge updates)
- **Offline banner**: Persistent top-banner when network is unavailable; dismisses on reconnect
