# User Story - US_067

## Story ID

* ID Format: US_067

## Story Title

* API Rate Limiting with Redis

## Description

* As a **system**, I want API rate limiting enforced via Redis-backed counters (100 req/min authenticated, 20 req/min unauthenticated), so that the system is protected from abuse and resource exhaustion.

## Acceptance Criteria

* Given an authenticated user, When they exceed 100 requests per minute, Then the system MUST return 429 Too Many Requests with a `Retry-After` header.
* Given an unauthenticated client, When they exceed 20 requests per minute, Then the system MUST return 429 Too Many Requests with a `Retry-After` header.
* Given rate limiting is active, When Redis stores the counters, Then counters MUST be distributed across API instances using Redis as shared state.
* Given rate limit thresholds are configurable, When an admin updates them via system settings, Then new thresholds MUST take effect without application restart.
* Given a rate-limited response is returned, When inspecting response headers, Then `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers MUST be present.

## Edge Cases

* What happens when Redis is unavailable? The system MUST fall back to in-memory rate limiting per instance and log a warning.
* How does the system handle rate limiting for admin users? Admin users MUST have a separate, higher rate limit threshold.
