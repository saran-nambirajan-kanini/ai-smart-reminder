# User Story - US_037

## Story ID

* ID Format: US_037

## Story Title

* Notification Retry and Fallback Logic

## Description

* As a **system**, I want notification delivery to retry up to 3 times with exponential backoff and attempt a fallback channel on primary failure, so that notification delivery is maximally reliable.

## Acceptance Criteria

* Given a notification delivery fails on the primary channel, When the retry policy activates, Then the system MUST retry up to 3 times with exponential backoff intervals of 1s, 4s, and 16s.
* Given all 3 retries fail on the primary channel, When the final retry fails, Then the system MUST attempt delivery via a fallback channel (e.g., push fails → try email).
* Given the fallback channel also fails, When all delivery attempts are exhausted, Then the Notification record MUST be updated to DeliveryStatus=Failed with a log entry detailing all attempts.
* Given a retry succeeds, When delivery is confirmed, Then the system MUST stop further retries and update DeliveryStatus to Sent/Delivered.
* Given the retry policy is executing, When logging retry attempts, Then each attempt MUST include the attempt number, channel, and error reason.

## Edge Cases

* What happens when the user has only one notification channel enabled and it fails? The system MUST mark as Failed after retry exhaustion since no fallback channel is available.
* How does the system handle a notification stuck in retry during application restart? Hangfire MUST persist the retry state and resume processing after restart.
