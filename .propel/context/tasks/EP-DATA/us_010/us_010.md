# User Story - US_010

## Story ID

* ID Format: US_010

## Story Title

* Entities: RefreshToken, Metrics, AIInteractionLog

## Description

* As a **developer**, I want EF Core entity configurations for RefreshToken, UserProductivityMetrics, and AIInteractionLog, so that authentication token management, analytics aggregation, and AI audit data can be persisted correctly.

## Acceptance Criteria

* Given the Domain layer exists, When the RefreshToken entity is defined, Then it MUST include: Id (GUID), UserId (FK), Token (hashed), ExpiresAt, RevokedAt, CreatedAt.
* Given the Domain layer exists, When the UserProductivityMetrics entity is defined, Then it MUST include: Id (GUID), UserId (FK), PeriodStart, PeriodEnd, PeriodType (Daily/Weekly/Monthly enum), TotalTasks, CompletedTasks, MissedReminders, OnTimeCompletionRate, ProductivityScore, MostProductiveDay, AverageCompletionTime, CreatedAt.
* Given the Domain layer exists, When the AIInteractionLog entity is defined, Then it MUST include: Id (GUID), UserId (FK), RequestType, PromptSummary, ResponseSummary, TokensUsed, LatencyMs, ModelVersion, CreatedAt.
* Given the RefreshToken entity is configured, When inspecting its EF Core configuration, Then the Token column MUST store a hashed value (not plaintext) and have an index for efficient lookups.
* Given all entities are configured, When running EF Core migrations, Then all three entities MUST be created as tables in the PostgreSQL database.

## Edge Cases

* What happens when a RefreshToken is revoked and then reused? The application MUST detect the revoked status and reject the token.
* How does the system handle UserProductivityMetrics for a period with zero tasks? All numeric fields MUST default to 0 and ProductivityScore to 0.0 rather than null.
