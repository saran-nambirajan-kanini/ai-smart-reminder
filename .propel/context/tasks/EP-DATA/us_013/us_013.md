# User Story - US_013

## Story ID

* ID Format: US_013

## Story Title

* Seed Data for Development and Testing

## Description

* As a **developer**, I want seed data for development and testing that populates all entity types with realistic sample data, so that frontend and backend development can proceed with representative data without manual setup.

## Acceptance Criteria

* Given the seed data is configured, When the database is initialized, Then it MUST create at least: 2 users (1 regular, 1 admin), 10 tasks with varying priorities and statuses, 5 categories, 5 reminders, 5 notifications, and 3 audit log entries.
* Given seed data includes users, When inspecting seeded user records, Then passwords MUST be properly hashed (not stored in plaintext) and emails MUST be unique.
* Given seed data includes tasks, When inspecting seeded task records, Then tasks MUST cover all priority levels (High, Medium, Low) and all status values (Active, Completed, Deleted).
* Given seed data is applied, When running the application in development mode, Then the dashboard and task list MUST display the seeded data correctly.
* Given the seed data configuration exists, When running in production mode, Then seed data MUST NOT be applied automatically.

## Edge Cases

* What happens when seed data is run on an already-seeded database? The seed operation MUST be idempotent, checking for existing data before inserting.
* How does the system handle seed data with FK references? Seed data MUST be inserted in dependency order (Users first, then Tasks, then Reminders, etc.).
