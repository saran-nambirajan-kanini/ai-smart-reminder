# User Story - US_008

## Story ID

* ID Format: US_008

## Story Title

* Core Domain Entities: User, Task, Category, Note

## Description

* As a **developer**, I want EF Core entity configurations for User, Task, Category, TaskCategory, and Note with attribute definitions, data types, and constraints, so that core domain data can be persisted with proper schema definitions.

## Acceptance Criteria

* Given the Domain layer exists, When the User entity is defined, Then it MUST include: Id (GUID), Name, Email (unique), PasswordHash, NotificationPreferences (JSON), QuietHoursStart, QuietHoursEnd, Role (User/Admin enum), OAuthProvider, OAuthId, IsActive, CreatedAt, UpdatedAt.
* Given the Domain layer exists, When the Task entity is defined, Then it MUST include: Id (GUID), UserId (FK), Title (required), Description, DueDate, Priority (High/Medium/Low enum, default Medium), Status (Active/Completed/Deleted enum), CompletionTimestamp, IsDeleted, DeletedAt, CreatedAt, UpdatedAt.
* Given the Domain layer exists, When the Category entity is defined, Then it MUST include: Id (GUID), UserId (FK), Name; with a unique constraint on (UserId, Name).
* Given the Domain layer exists, When the Note entity is defined, Then it MUST include: Id (GUID), TaskId (FK), Content (required), CreatedAt, UpdatedAt.
* Given the TaskCategory join entity exists, When inspecting it, Then it MUST define a composite key of TaskId (FK) and CategoryId (FK) enabling many-to-many relationships.

## Edge Cases

* What happens when a User email exceeds the maximum length? Entity configuration MUST enforce a maximum length constraint (e.g., 256 characters) at the database level.
* How does the system handle a Task created without a DueDate? The DueDate column MUST be nullable, and the application MUST not require it.
