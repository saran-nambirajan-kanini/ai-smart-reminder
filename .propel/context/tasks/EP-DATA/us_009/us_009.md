# User Story - US_009

## Story ID

* ID Format: US_009

## Story Title

* Entities: Reminder, Notification, AuditLog

## Description

* As a **developer**, I want EF Core entity configurations for Reminder, Notification, and AuditLog with all required attributes and data types, so that reminder scheduling, notification delivery tracking, and audit compliance data can be persisted correctly.

## Acceptance Criteria

* Given the Domain layer exists, When the Reminder entity is defined, Then it MUST include: Id (GUID), TaskId (FK), UserId (FK), ScheduledTime (UTC timestamptz), FrequencyType (None/Daily/Weekly/Monthly/Custom enum), CustomIntervalDays, IsActive, LastTriggeredAt, NextOccurrence, SnoozedUntil, CreatedAt, UpdatedAt.
* Given the Domain layer exists, When the Notification entity is defined, Then it MUST include: Id (GUID), UserId (FK), ReminderId (FK), Channel (Push/Email/SMS enum), Content, DeliveryStatus (Pending/Sent/Delivered/Failed enum), IsRead, SentAt, CreatedAt.
* Given the Domain layer exists, When the AuditLog entity is defined, Then it MUST include: Id (GUID), UserId (FK), Action, EntityType, EntityId, Details (JSON/jsonb column), IpAddress, Timestamp (UTC timestamptz).
* Given the AuditLog entity is configured, When inspecting its EF Core configuration, Then the entity MUST be configured as append-only (no update or delete operations permitted at the repository level).
* Given all entities are configured, When inspecting timestamp columns, Then all date/time fields MUST use PostgreSQL `timestamptz` type storing UTC values.

## Edge Cases

* What happens when a Notification is created with an invalid Channel value? EF Core MUST reject the insert with a validation error indicating the invalid enum value.
* How does the system handle a Reminder with both FrequencyType=Custom and CustomIntervalDays=null? A domain validation rule MUST require CustomIntervalDays when FrequencyType is Custom.
