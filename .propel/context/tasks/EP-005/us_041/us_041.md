# User Story - US_041

## Story ID

* ID Format: US_041

## Story Title

* Azure OpenAI Service Integration Setup

## Description

* As a **developer**, I want Azure OpenAI Service integrated as the primary AI provider with configurable model selection and OpenAI API as fallback, so that AI features have a reliable LLM backend with provider-swappable architecture.

## Acceptance Criteria

* Given the application configuration, When AI service settings are provided, Then the system MUST register an IAiService interface with AzureOpenAiService as the primary implementation.
* Given Azure OpenAI is configured, When an AI request is made, Then the system MUST use the configured model deployment (GPT-4o-mini for classification, GPT-4o for insights) with JSON structured output mode.
* Given the AI provider configuration, When model deployment names and API keys are specified, Then the system MUST validate the configuration at startup and fail with a descriptive error if invalid.
* Given the IAiService interface, When inspecting its contract, Then it MUST define methods for NLP parsing, categorization, prioritization, and insight generation returning structured DTOs.
* Given the fallback provider is configured, When Azure OpenAI is unavailable and circuit breaker opens, Then the system MUST route requests to the OpenAI API fallback.

## Edge Cases

* What happens when both AI providers are unavailable? The system MUST fall back to deterministic processing without AI features (per AIR-004).
* How does the system handle invalid API keys? The system MUST log the authentication error and immediately open the circuit breaker for that provider.
