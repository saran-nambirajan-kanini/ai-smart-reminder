# User Story - US_019

## Story ID

* ID Format: US_019

## Story Title

* User Profile View and Update

## Description

* As an **authenticated user**, I want to view and update my profile information (name, email, notification preferences), so that I can keep my account details current and control how I receive notifications.

## Acceptance Criteria

* Given an authenticated user calls the profile endpoint, When the request is valid, Then the system MUST return the user's name, email, notification preferences, quiet hours, and OAuth provider linkage status.
* Given an authenticated user submits profile updates, When the name is changed, Then the system MUST update the name and return 200 OK with the updated profile.
* Given an authenticated user updates their email, When the new email is already in use by another account, Then the system MUST return 409 Conflict with "email already in use" error.
* Given an authenticated user updates notification preferences, When the preferences are valid, Then the system MUST save per-channel (push, email, SMS) enable/disable settings and quiet hours.
* Given the profile is updated, When inspecting the database record, Then the UpdatedAt timestamp MUST be refreshed.

## Edge Cases

* What happens when a user tries to update their password through the profile endpoint? The profile endpoint MUST NOT accept password changes; password updates MUST use the dedicated password reset flow.
* How does the system handle a user trying to remove their email entirely? The system MUST reject the update as email is a required field.
